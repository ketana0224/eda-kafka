## アーキテクチャ概要

2.Orchestration.md と出来る限り同じワークフローを **Azure Logic Apps Consumption** で構成する。

### プラン選択理由: Logic Apps Consumption

| 観点 | **Consumption（採用）** | Standard（断念） |
|---|---|---|
| ステートフルワークフロー | ○ | ○ |
| Apache Kafka Managed Connector | **✗（Managed Connector なし）** | ✓（組み込み） |
| Kafka 統合方式 | **Azure Functions ブリッジ（HTTP ↔ Kafka）** | 組み込みコネクタ（設定のみ）|
| `allowSharedKeyAccess: false` 対応 | **✓ ストレージ不要** | ✗ Azure Files 必須 |
| Managed Identity で Cosmos DB 接続 | ✓（HTTP + MSI） | ✓ |
| コールバック URL 保存先 | Cosmos DB for NoSQL（MSI）| Azure Table Storage（Shared Key）|
| ホスティング | マルチテナント | WS1（Azure Files 必須）|

> **Cosmos DB 採用理由**: `allowSharedKeyAccess: false` ポリシーが全ストレージアカウントに適用されており、
> Azure Table Storage Managed Connector（Shared Key 使用）が使えない。
> Cosmos DB は MSI（Bearer token + `https://cosmos.azure.com`）で REST API を直接呼び出せる。

> **Kafka 統合方式について**: Logic Apps Consumption は Apache Kafka の Managed Connector を持たない。
> Kafka との連携は `func-ketana-ext2-kafka-bridge`（Azure Functions）をブリッジとして使用する。
> - **KafkaPublisherFunction**: Logic App からの HTTP POST を受け取り Kafka にパブリッシュ
> - **KafkaConsumerFunctions**: Kafka トピックをポーリングし、Logic App の HTTP trigger を呼び出す

→ `allowSharedKeyAccess: false` の制約に対応し、Kafka 統合は Azure Functions ブリッジで実現できる **Consumption** を採用する。

### Durable Functions との主な差分

| 観点 | Durable Functions（2.Orchestration.md） | Logic Apps Consumption（本仕様） |
|---|---|---|
| オーケストレーター実装 | Java コード（Durable Task SDK） | ノーコード ワークフロー JSON |
| Kafka 統合 | KafkaTrigger バインディング（コード必要） | Azure Functions ブリッジ（HTTP ↔ Kafka） |
| 状態管理バックエンド | Azure Durable Task Scheduler | Logic Apps 組み込み（Azure Storage） |
| 相関メカニズム | `waitForExternalEvent` + `raiseEvent` | HTTP Webhook アクション + Cosmos DB コールバック URL |
| タイムアウト | `createTimer(30min)` | HTTP Webhook タイムアウト（PT30M） |
| ダッシュボード | Durable Task Scheduler Dashboard | Azure Portal 実行履歴 / Application Insights |
| 既知の問題 | `cardinality` 手動追加・KafkaTrigger ラッパー JSON 解析 | Kafka ブリッジ Function App の追加管理が必要 |

### ワークフロー構成

Logic Apps Consumption は **1 リソース = 1 ワークフロー**。以下の 5 リソースを作成する。

| Logic App リソース名 | トリガー | 役割 |
|---|---|---|
| `la-ketana-ext2-callback-reg` | HTTP | コールバック URL を Cosmos DB に Upsert |
| `la-ketana-ext2-order-saga` | HTTP | Saga の指揮者（在庫→配送→確認） |
| `la-ketana-ext2-inv-reserved` | HTTP（Functions ブリッジ経由） | 在庫引当成功イベント処理 |
| `la-ketana-ext2-inv-failed` | HTTP（Functions ブリッジ経由） | 在庫不足イベント処理 |
| `la-ketana-ext2-ship-sched` | HTTP（Functions ブリッジ経由） | 配送手配完了イベント処理 |

> **Kafka ブリッジ補足**: Logic Apps Consumption は Apache Kafka Managed Connector を持たない。
> Kafka との連携は `func-ketana-ext2-kafka-bridge`（Azure Functions）をブリッジとして使用する。
> - `KafkaPublisherFunction`: Logic App からの HTTP POST を受け取り Kafka にパブリッシュ（Publish 系アクション用）
> - `KafkaConsumerFunctions`: Kafka トピックをポーリングし、対応する Logic App Consumer の HTTP trigger を呼び出す

### OrderSagaOrchestrator フロー

```
[HTTP Client]
     │ POST /api/OrderSagaOrchestrator/triggers/manual/invoke
     ▼
[HTTP Trigger — OrderSagaOrchestrator (Stateful)]
     │ orchestrationId = @{workflow().run.name}  ← 相関 ID として使用
     │
     ├─① HTTP アクション → OrderService (ACA)  ──▶ PostgreSQL(order)
     │     └─ orderId を取得（PENDING で DB 保存）
     │
     ├─② HTTP アクション → KafkaPublisherFunction → Kafka topic: inventory.reserve.command
     │     └─ payload に workflowRunId を埋め込む
     │
     ├─③ HTTP Webhook アクション（waitForCallback）
     │     ├─ Subscribe: POST /api/CallbackRegistrar（Cosmos DB に callbackUrl を登録）
     │     body: {"orchestrationId": "@{concat(workflow().run.name, '-inv')}", "callbackUrl": "@{listCallbackUrl()}"}
     │     └─ タイムアウト: PT30M（タイムアウト時は補償フローへ）
     │
     │   ─── 在庫引当成功（KafkaConsumerFunction が検知 → la-ketana-ext2-inv-reserved を HTTP 呼出） ──────────
     │
     ├─④ HTTP アクション → KafkaPublisherFunction → Kafka topic: shipping.schedule.command
     │     └─ payload に workflowRunId を埋め込む
     │
     ├─⑤ HTTP Webhook アクション（waitForCallback）
     │     ├─ Subscribe: POST /api/CallbackRegistrar
     │     └─ タイムアウト: PT30M
     │
     ├─⑥ HTTP アクション → KafkaPublisherFunction → Kafka topic: order.confirmed
     │           ↓
     │   [OrderService Consumer（マイクロサービス）] → status = CONFIRMED
     └─ 完了 (Succeeded)

   ─── 在庫不足 / タイムアウト（補償フロー）──────────────────────────────────────────
     ├─⑦ HTTP アクション → KafkaPublisherFunction → Kafka topic: inventory.release.command
     │           ↓
     │   [InventoryService] → 引当解放
     ├─⑧ HTTP アクション → KafkaPublisherFunction → Kafka topic: order.cancelled
     │           ↓
     │   [OrderService Consumer（マイクロサービス）] → status = CANCELLED
     └─ 完了 (Succeeded with compensation)
```

### 相関メカニズム（HTTP Webhook ↔ Kafka）

`OrderSagaOrchestrator` は `@{workflow().run.name}` を `orchestrationId` として Kafka コマンドに埋め込む。  
各マイクロサービスはレスポンスイベントに `orchestrationId` をそのまま返す（パススルー）。  
`KafkaConsumerFunctions`（Azure Functions）が Kafka トピックをポーリングしてレスポンスを受信し、  
`orchestrationId` をキーに対応する Logic App Consumer（`la-ketana-ext2-inv-reserved` 等）の HTTP trigger を呼び出す。  
Logic App Consumer は Cosmos DB からコールバック URL を取得し、HTTP POST でオーケストレーターを再開する。

**コールバック URL 登録フロー（③⑤の Webhook アクション）:**

1. `la-ketana-ext2-order-saga` の HTTP Webhook アクションが `@{listCallbackUrl()}` でコールバック URL を取得
2. Subscribe 呼び出しで `la-ketana-ext2-callback-reg` に `{orchestrationId, callbackUrl}` を送信  
   ※ orchestrationId は `{run.name}-inv`（在庫）/ `{run.name}-ship`（配送）のサフィックスで区別
3. `la-ketana-ext2-callback-reg` が **Cosmos DB** にドキュメントを Upsert  
   `{id: orchestrationId, orchestrationId, callbackUrl}` (TTL: 7200 秒)

**Kafka → Logic App 呼び出しフロー（Azure Functions ブリッジ）:**

1. `KafkaConsumerFunctions`（Azure Functions）が Kafka トピックをポーリングしてメッセージを受信
2. メッセージから `orchestrationId` を抽出し、対応する Logic App Consumer の HTTP trigger を POST 呼び出す
3. Logic App Consumer（`la-ketana-ext2-inv-reserved` 等）が HTTP trigger でイベントを受信
4. Cosmos DB REST API (GET) で `orchestrationId` をキーにコールバック URL を取得（MSI 認証）
5. HTTP POST でコールバック URL を呼び出してオーケストレーターを再開
6. Cosmos DB REST API (DELETE) でドキュメントを削除（クリーンアップ）

参照: https://learn.microsoft.com/ja-jp/azure/logic-apps/logic-apps-http-endpoint

### Phase 0 — 前提確認

```powershell
$cfg = Import-PowerShellDataFile .\infra\config.psd1
$laRG = 'rg-ketana-ext2-logicapp'  # Logic Apps 専用 RG
az account set --subscription $cfg.SubscriptionId

# Logic Apps 専用 RG 作成（初回のみ）
az group create --name $laRG --location $cfg.Location

# Kafka 公開 IP 取得
$kafkaIp = az container show -n $cfg.ContainerGroup -g $cfg.ResourceGroup `
  --query ipAddress.ip -o tsv
Write-Host "kafkaIp=$kafkaIp"

# Kafka 稼働確認
az container show -n $cfg.ContainerGroup -g $cfg.ResourceGroup `
  --query instanceView.state -o tsv
# 期待値: Running

# ACA マイクロサービス確認
foreach ($name in @("aca-order","aca-inventory","aca-shipping")) {
  az containerapp show -n $name -g $cfg.ResourceGroup --query properties.runningStatus -o tsv
}
# 期待値: Running（×3）

# Kafka ブリッジ用 ブートストラップサーバー設定
$kafkaBootstrapServers = "${kafkaIp}:9092"
Write-Host "kafkaBootstrapServers=$kafkaBootstrapServers"

# リソースプロバイダー登録（初回のみ / 未登録の場合のみ実行）
$providers = @('Microsoft.DocumentDB','Microsoft.Logic','Microsoft.Web',
               'Microsoft.Storage','Microsoft.App')
foreach ($p in $providers) {
    $state = az provider show -n $p --query registrationState -o tsv
    if ($state -ne 'Registered') {
        Write-Host "Registering $p ..."
        az provider register --namespace $p
    } else {
        Write-Host "$p : $state"
    }
}
# 登録完了を待つ（登録したプロバイダーがある場合のみ必要）
foreach ($p in $providers) {
    do {
        $state = az provider show -n $p --query registrationState -o tsv
        Write-Host "$(Get-Date -Format HH:mm:ss) $p : $state"
        if ($state -ne 'Registered') { Start-Sleep 15 }
    } while ($state -ne 'Registered')
}
Write-Host "全プロバイダー登録済み"
```

```powershell
# 1-3d. Logic Apps デプロイ（az rest で各ワークフロー定義を PUT）
#       ワークフロー定義 JSON: logicapp/consumption/workflows/*.json
$laBaseUri = "https://management.azure.com/subscriptions/$($cfg.SubscriptionId)" +
             "/resourceGroups/$laRG/providers/Microsoft.Logic/workflows"
$laApiVer  = "2019-05-01"
$wfDir     = "logicapp\consumption\workflows"
$kafkaBridgePublishUrl = "https://func-ketana-ext2-kafka-bridge.azurewebsites.net/api/kafka/publish"

function Deploy-LogicApp([string]$Name, [string]$DefFile, [hashtable]$Params) {
    $def  = Get-Content $DefFile -Raw | ConvertFrom-Json
    $body = @{
        location   = $cfg.Location
        identity   = @{ type = 'SystemAssigned' }
        properties = @{ state = 'Enabled'; definition = $def; parameters = $Params }
    } | ConvertTo-Json -Depth 30 -Compress
    $tmpFile = [System.IO.Path]::GetTempFileName()
    $body | Set-Content $tmpFile -Encoding utf8
    az rest --method PUT --uri "${laBaseUri}/${Name}?api-version=$laApiVer" `
      --body "@$tmpFile" --headers 'Content-Type=application/json'
    Remove-Item $tmpFile -Force
    Write-Host "  deployed: $Name"
}

Deploy-LogicApp 'la-ketana-ext2-callback-reg' "$wfDir\callback-reg.json" @{ cosmosDbEndpoint=@{value=$cosmosEndpoint} }
Deploy-LogicApp 'la-ketana-ext2-inv-reserved' "$wfDir\inv-reserved.json" @{ cosmosDbEndpoint=@{value=$cosmosEndpoint} }
Deploy-LogicApp 'la-ketana-ext2-inv-failed'   "$wfDir\inv-failed.json"   @{ cosmosDbEndpoint=@{value=$cosmosEndpoint} }
Deploy-LogicApp 'la-ketana-ext2-ship-sched'   "$wfDir\ship-sched.json"   @{ cosmosDbEndpoint=@{value=$cosmosEndpoint} }

    kafkaPublisherUrl    = @{ value = $kafkaBridgePublishUrl }

}
# 期待値: 各 Logic App が "provisioningState":"Succeeded" で作成される
```

### Phase 1.5 — Kafka ブリッジ Function App コードビルド & デプロイ

> **ソースコード**: `services/saga-orchestrator/` 内の以下のクラスが Kafka ブリッジとして機能する。
> - `KafkaPublisherFunction`: `POST /api/kafka/publish` → Kafka トピックに発行（Logic App から呼び出し）
> - `KafkaConsumerFunctionsLA`: Kafka トピックをポーリング → 対応する Logic App Consumer の HTTP trigger を呼び出す（**Durable Functions 依存なし**）
>
> **⚠️ `-Pbridge` プロファイル必須**  
> `services/saga-orchestrator/pom.xml` には Durable Functions Saga 用の設定（`host.json` に `durableTask` セクション）がデフォルトで含まれる。  
> `func-ketana-ext2-kafka-bridge` は Durable 依存がないため、**`-Pbridge` プロファイル**を指定することで `host-bridge.json`（`durableTask` セクションなし）を使用する。  
> `-Pbridge` を省略すると Function App ホスト起動時に `DURABLE_TASK_SCHEDULER_CONNECTION_STRING` 不在エラーが発生し、全関数が登録されない。

### Phase 2.5 — Consumer Logic App Trigger URL → Function App 設定

> Kafka Consumer Functions が受信イベントを正しい Logic App Consumer に転送するために、


```powershell
# 2.5-2. Function App の app settings を更新（URL に & が含まれるため REST API + temp file 方式）
$funcName = 'func-ketana-ext2-kafka-bridge'
$funcUri  = "https://management.azure.com/subscriptions/$($cfg.SubscriptionId)" +
            "/resourceGroups/$laRG/providers/Microsoft.Web/sites/$funcName" +
            "/config/appsettings?api-version=2022-03-01"

az rest --method PUT --uri $funcUri --body "@$tmpFile" --headers 'Content-Type=application/json' `
  --query "properties.LOGIC_APP_INV_RESERVED_URL" -o tsv
```
