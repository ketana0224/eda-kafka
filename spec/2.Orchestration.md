## アーキテクチャ概要

Orchestration Saga の指揮者として **Azure Durable Functions** を採用する。  
HTTP の入口は Durable Functions の HTTP Trigger とし、クライアントは直接 Orchestrator を呼び出す。  
OrderService / InventoryService / ShippingService は **Azure Container Apps** にデプロイし、Orchestrator と Kafka（ACI）経由で非同期連携する。

```
[HTTP Client]
     │ POST /api/orders
     ▼
[HTTP Trigger Function (func-ketana-ext2-saga-orch)]
     │ startNewAsync(instanceId)
     ▼
[Durable Functions Orchestrator]
     │
     ├─① Activity: CreateOrder
     │     │ HTTP POST → [OrderService (ACA)] ──▶ PostgreSQL(order)
     │     │             (PENDING で DB 保存・orderId 返却)
     │     ▼
     ├─② Activity: PublishInventoryReserveCommand
     │     │ Kafka: inventory.reserve.command（orchestrationId 付き）
     │     │               ▼
     │     │   [InventoryService (ACA)] ──▶ PostgreSQL(inventory)
     │     │               │ Kafka: inventory.reserved          （orchestrationId 付き）
     │     │               │       inventory.reservation.failed （orchestrationId 付き）
     │     │               ▼
     │     │   [KafkaConsumer Function]
     │     │               │ raiseEvent(orchestrationId, "InventoryResult", payload)
     │     ▼
     ├─③ waitForExternalEvent("InventoryResult") or createTimer(30min)
     │
     │   ─── 在庫引当成功 ─────────────────────────────────────────────────────────
     │
     ├─④ Activity: PublishShippingScheduleCommand
     │     │ Kafka: shipping.schedule.command（orchestrationId 付き）
     │     │               ▼
     │     │   [ShippingService (ACA)] ──▶ PostgreSQL(shipping)
     │     │               │ Kafka: shipping.scheduled（orchestrationId 付き）
     │     │               ▼
     │     │   [KafkaConsumer Function]
     │     │               │ raiseEvent(orchestrationId, "ShippingResult", payload)
     │     ▼
     ├─⑤ waitForExternalEvent("ShippingResult") or createTimer(30min)
     │
     ├─⑥ Activity: PublishOrderConfirmed
     │     │ Kafka: order.confirmed
     │     │               ▼
     │     │   [OrderService Consumer] → status = CONFIRMED
     │     ▼
     └─ Orchestration 完了 (Completed)

   ─── 在庫不足 / タイムアウト時（補償フロー）────────────────────────────────────
     ├─⑦ Activity: PublishInventoryReleaseCommand
     │     │ Kafka: inventory.release.command ──▶ InventoryService（引当解放）
     ├─⑧ Activity: PublishOrderCancelled
     │     │ Kafka: order.cancelled
     │     │               ▼
     │     │   [OrderService Consumer] → status = CANCELLED
     └─ Orchestration 完了 (Completed with compensation)
```

### 相関メカニズム（waitForExternalEvent ↔ Kafka）

Orchestrator が Kafka コマンドを発行する際、メッセージに `orchestrationId`（= Durable Functions の `instanceId`）を埋め込む。  
各マイクロサービスはレスポンスイベントにその `orchestrationId` をそのまま返す（パススルー）。  
Kafka Consumer Function がレスポンスを受信し、`orchestrationId` をキーに `DurableTaskClient.raiseEvent()` を呼び出すことで、対応する Orchestrator インスタンスの `waitForExternalEvent` が解除される。

Durable Functions の `waitForExternalEvent` で Kafka 応答イベントを待機し、`createTimer` でタイムアウト自動キャンセルを実現する。

参照: https://learn.microsoft.com/en-us/azure/architecture/patterns/saga

### Phase 3 — データストアの作成

```powershell
# PostgreSQL Flexible Server × 3（各サービス専用）

# パスワードをセキュアに入力（スクリプト実行時に一度だけ入力。ファイルには保存しない）
# 実際のパスワード: Chinami02241017
$pgPasswordSecure = Read-Host "PostgreSQL admin password" -AsSecureString
$pgPassword = [System.Net.NetworkCredential]::new("", $pgPasswordSecure).Password

$pgServices = @(
  @{ Name = "psql-ketana-ext2-order";     Db = "orderdb"     },
  @{ Name = "psql-ketana-ext2-inventory"; Db = "inventorydb" },
  @{ Name = "psql-ketana-ext2-shipping";  Db = "shippingdb"  }
)

foreach ($pg in $pgServices) {
  Write-Host "$(Get-Date -Format 'HH:mm:ss')  サーバー作成: $($pg.Name)"
  az postgres flexible-server create `
    --resource-group $cfg.ResourceGroup `
    --name $pg.Name `
    --location $cfg.Location `
    --admin-user sagaadmin `
    --admin-password $pgPassword `
    --sku-name Standard_B1ms `
    --tier Burstable `
    --public-access 0.0.0.0 `
    --output table

  # --database-name は flexible-server create では使用不可（CLI の仕様変更）
  # サーバー作成後に別コマンドで DB を作成する
  Write-Host "$(Get-Date -Format 'HH:mm:ss')  DB作成: $($pg.Db)"
  az postgres flexible-server db create `
    --resource-group $cfg.ResourceGroup `
    --server-name $pg.Name `
    --database-name $pg.Db `
    --output table

  Write-Host "$(Get-Date -Format 'HH:mm:ss')  完了: $($pg.Name) / $($pg.Db)"
}
```

> **⚠️ 注意**: `--public-access 0.0.0.0` は検証環境向け設定（全 IP からのアクセスを許可）。本番環境では VNet 統合またはプライベートエンドポイントを使用すること。

### Phase 4 — Orchestrator (Durable Functions) の作成

> **このサブスクリプションの制約（実施時に判明）**:
> - Azure Policy により `allowSharedKeyAccess: false` が強制される → Shared Key / SAS 接続文字列は使用不可
> - Consumption / ElasticPremium プランのクォータが 0 → 作成不可
> - **B1 App Service Plan は japanwest で作成可能**（japaneast は Standard 系クォータ不足の可能性あり）
> - System-Assigned MI 有効化時に旧リージョン（japaneast）の SP が AAD に残存し `AlreadyExistServicePrincipalInDifferentRegion` エラーが発生する可能性がある → Function App 名を変えることで回避
> - **対策**: B1 App Service Plan（japanwest）で Function App を作成し、Storage は Managed Identity で接続する
> - Flex Consumption は `az functionapp restart` が効かない・`admin/functions` が常に空など運用上の問題があるため不採用

> **💡 推奨順序**: Azure Portal の Function App 作成ウィザードでは「Durable Functions」タブから DTS を選択・新規作成できる。
> CLI でも DTS → Function App の順で作成すると `az functionapp create` 時にエンドポイントを設定に含められる（後から `appsettings set` 不要）。
> 以下の手順はその順序で記述する。

```powershell
# Microsoft.Web / Microsoft.DurableTask プロバイダーを登録
az provider register --namespace Microsoft.Web --wait
az provider register --namespace Microsoft.DurableTask --wait
az provider show --namespace Microsoft.Web --query registrationState -o tsv
# 期待値: Registered

# CLI 拡張インストール
az extension add --name durabletask --yes

# Storage Account（AzureWebJobsStorage 用・DTS への状態ストア移行後も必要）
# allowSharedKeyAccess は Policy で false に固定されるため --allow-shared-key-access は指定しない
az storage account create `
  --name "stketanaext2saga" `
  --resource-group $cfg.ResourceGroup `
  --location $cfg.Location `
  --sku Standard_LRS

# App Service Plan（B1 / japanwest）
az appservice plan create `
  --name "asp-ketana-ext2-saga" `
  --resource-group $cfg.ResourceGroup `
  --location "japanwest" `
  --sku B1 `
  --is-linux

# Durable Task Scheduler 作成（最大 15 分・Function App より先に作成してエンドポイントを確定させる）
az durabletask scheduler create `
  --name "dts-ketana-ext2-saga" `
  --resource-group $cfg.ResourceGroup `
  --location "japanwest" `
  --ip-allowlist "[0.0.0.0/0]" `
  --sku-name "dedicated" `
  --sku-capacity "1" `
  --output table
# 期待値: ProvisioningState = Succeeded

# Task Hub 作成
az durabletask taskhub create `
  --resource-group $cfg.ResourceGroup `
  --scheduler-name "dts-ketana-ext2-saga" `
  --name "sagahub" `
  --output table

$schedulerEndpoint = az durabletask scheduler show `
  --name "dts-ketana-ext2-saga" -g $cfg.ResourceGroup `
  --query properties.endpoint -o tsv
Write-Host "endpoint: $schedulerEndpoint"

# Function App 作成（DTS エンドポイントを初期設定に含める）
# 注意: --plan で App Service Plan を指定する（--consumption-plan-location は不要）
# 注意: --runtime-version は "21.0" と指定する
# 注意: --app-insights で既存の App Insights を指定すると自動生成が抑止される（後処理不要）
az functionapp create `
  --resource-group $cfg.ResourceGroup `
  --name "func-ketana-ext2-saga-orch" `
  --plan "asp-ketana-ext2-saga" `
  --runtime java `
  --runtime-version 21.0 `
  --functions-version 4 `
  --storage-account "stketanaext2saga" `
  --os-type Linux `
  --app-insights "appi-ketana-ext2-eda-kafka" `
  --app-settings `
    "DURABLE_TASK_SCHEDULER_CONNECTION_STRING=Endpoint=${schedulerEndpoint};Authentication=ManagedIdentity" `
    "TASKHUB_NAME=sagahub"

# System-Assigned Managed Identity を有効化
$principalId = az functionapp identity assign `
  --name "func-ketana-ext2-saga-orch" `
  --resource-group $cfg.ResourceGroup `
  --query principalId -o tsv

$storageId = az storage account show `
  --name "stketanaext2saga" `
  --resource-group $cfg.ResourceGroup `
  --query id -o tsv

$taskHubId = az durabletask taskhub show `
  --resource-group $cfg.ResourceGroup `
  --scheduler-name "dts-ketana-ext2-saga" `
  --name "sagahub" `
  --query id -o tsv

# ⚠️ 重要 — ロール付与は AzureWebJobsStorage__credential=managedidentity の設定より必ず先に行うこと
# ロールが未付与の状態で MI 接続に切り替えると、Function App が Storage に一切アクセスできず
# 起動失敗（Application run failed）になる。これが System-Assigned MI のロールがゼロの状態で
# 発生した起動失敗の根本原因である。
# 以下の 5 ロール（Storage × 3 + DTS × 2）をすべて付与してから appsettings set を実行すること。

# Storage RBAC ロール付与（AzureWebJobsStorage 用の 3 ロール）
# Storage Blob Data Owner / Storage Queue Data Contributor / Storage Table Data Contributor
$roles = @(
  "b7e6dc6d-f1e8-4753-8033-0f276bb0955b",
  "974c5e8b-45b9-4653-ba55-5f855dd0fb88",
  "0a9a7e1f-b9d0-4cc4-a60d-0319b160aaa3"
)
foreach ($roleId in $roles) {
  az role assignment create `
    --assignee-object-id $principalId `
    --assignee-principal-type ServicePrincipal `
    --role $roleId `
    --scope $storageId `
    --output none
}

# Durable Task Worker ロール付与（ワーカー実行権限）
az role assignment create `
  --assignee-object-id $principalId `
  --assignee-principal-type ServicePrincipal `
  --role "Durable Task Worker" `
  --scope $taskHubId `
  --output none

# Durable Task Data Contributor ロール付与（スケジューリング権限・HTTP 202 を返すために必要）
# ⚠️ このロールがないと "PermissionDenied" エラーになり Orchestration が起動しない
az role assignment create `
  --assignee-object-id $principalId `
  --assignee-principal-type ServicePrincipal `
  --role "Durable Task Data Contributor" `
  --scope $taskHubId `
  --output none

# AzureWebJobsStorage を Managed Identity 接続に設定
# az functionapp create が自動設定した Shared Key 接続文字列を MI 接続に上書き・削除する
az functionapp config appsettings set `
  --name "func-ketana-ext2-saga-orch" `
  --resource-group $cfg.ResourceGroup `
  --settings `
    "AzureWebJobsStorage__accountName=stketanaext2saga" `
    "AzureWebJobsStorage__credential=managedidentity" `
  --output none

az functionapp config appsettings delete `
  --name "func-ketana-ext2-saga-orch" `
  --resource-group $cfg.ResourceGroup `
  --setting-names "AzureWebJobsStorage" `
  --output none

# アプリ設定（Kafka・タイムアウト）
# ORDER_SERVICE_URL は Phase 5 で aca-order をデプロイした後に設定する（後述）
az functionapp config appsettings set `
  --name "func-ketana-ext2-saga-orch" `
  --resource-group $cfg.ResourceGroup `
  --settings `
    "KAFKA_BOOTSTRAP_SERVERS=${kafkaIp}:9092" `
    "SAGA_TIMEOUT_MINUTES=30"
```

#### 4b — saga-orchestrator のビルドとデプロイ

App Service Plan（B1）は `mvn azure-functions:deploy` を使用する。  
`allowSharedKeyAccess: false` の影響を受けない（Kudu が直接 zip を受け取るため）。

> **⚠️ 注意**: `func azure functionapp publish --java` は使用しないこと。  
> このコマンドは内部で Node.js の `archiver` ライブラリが生成した zip を送信するが、
> そのフォーマット（zip64 拡張など）を Azure Functions Linux ホストが正常にマウントできず
> 「0 functions found」になる既知の問題がある。  
> **採用した方式**: `mvn package -DskipTests` で JAR を再生成 → `function.json` に `cardinality: ONE` を手動追加 → `mvn azure-functions:deploy` でデプロイする。

> **📝 KafkaTrigger の挙動（実施時に判明）**:  
> `@KafkaTrigger(cardinality = Cardinality.ONE, dataType = "string")` + `String` パラメータの場合、  
> KafkaTrigger はペイロードをそのまま渡すのではなく、以下のラッパー JSON を文字列として渡す。
> ```json
> {"Offset":12345,"Partition":0,"Topic":"inventory.reserved","Timestamp":"...","Value":"{...}","Headers":[]}
> ```
> `Value` フィールドが実際のメッセージペイロード（エスケープ済み JSON 文字列）。  
> `MAPPER.readValue(message, XxxMsg.class)` を直接呼ぶと `"Offset"` フィールドが未知扱いになり  
> `UnrecognizedPropertyException` が発生する。  
> **対応**: 各 Consumer で `extractValue(message)` を呼び出して `Value` を取り出してから `readValue` する。

**Step 1: Maven ビルド（JAR 再生成 + ステージングディレクトリ生成）**

> **⚠️ 既知の注意事項（実施時に判明）**:
> - `mvn compile` はクラスファイル（`.class`）を更新するが、JAR は更新しない。  
>   `mvn azure-functions:package` は既存 JAR をステージングにコピーするだけのため、  
>   `compile` → `azure-functions:package` のシーケンスでは**旧コードがデプロイされる**。  
>   **必ず `mvn package -DskipTests` で JAR を再生成してからデプロイすること。**
> - `azure-functions-maven-plugin` の既知バグにより、`@KafkaTrigger(cardinality = Cardinality.ONE)` を  
>   ソースに記述しても、`mvn package` / `azure-functions:package` 実行後の `function.json` に  
>   `"cardinality"` が出力されない。  
>   `cardinality` が `MANY`（デフォルト）のままだと KafkaConsumer がバッチ配列を受け取り  
>   デシリアライズ失敗する。**package 後に手動で追加する必要がある（Step 2 参照）。**

```powershell
Set-Location C:\GitHub\eda-kafka\services\saga-orchestrator

$mvn = "C:\tools\maven\apache-maven-3.9.6\bin\mvn.cmd"

# ★ package -DskipTests で JAR を再生成する（compile 単体では JAR は更新されない）
& $mvn package -DskipTests
# 期待値: BUILD SUCCESS
```

**Step 2: cardinality: ONE を function.json に手動追加**

> `azure-functions-maven-plugin` のバグにより `"cardinality"` が function.json に出力されないため、
> package 実行後に毎回以下で手動追加する。`azure-functions:deploy` は再 package しないため、
> 追加後の状態でデプロイされる。

```powershell
$funcBase = "target\azure-functions\func-ketana-ext2-saga-orch"
foreach ($fn in @("InventoryReservedConsumer","InventoryReservationFailedConsumer","ShippingScheduledConsumer")) {
    $path = "$funcBase\$fn\function.json"
    $content = Get-Content $path -Raw
    if ($content -notmatch '"cardinality"') {
        $content = $content -replace '("brokerList"\s*:\s*"%KAFKA_BOOTSTRAP_SERVERS%")', "`$1,`n    `"cardinality`" : `"ONE`""
        Set-Content $path $content -NoNewline
        Write-Host "cardinality 追加: $fn"
    } else {
        Write-Host "cardinality 既存: $fn"
    }
}

# 確認
Get-Content "$funcBase\InventoryReservedConsumer\function.json" | Select-String "cardinality"
# 期待値: "cardinality" : "ONE"
```

#### シナリオ 11 — 補償フロー（在庫不足）


4. `orderdb.orders` テーブルの最新レコードで `status = CANCELLED` を確認
